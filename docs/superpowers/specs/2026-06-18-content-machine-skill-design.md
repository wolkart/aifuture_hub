# Дизайн скилла `content-machine` (тонкий оркестратор-делегат)

Дата: 2026-06-18
Статус: утверждён, готов к плану реализации (writing-plans)

## 1. Цель

Единая входная дверь «идея → готовый контент». Скилл определяет источник идеи, прогоняет его через готовые скиллы хаба и выдаёт финальный артефакт (рилс-сценарий и/или карусель). Сам контент-ремесло НЕ делает — **делегирует**; владеет только маршрутизацией, чекпойнтами и хэндоффами. DRY.

## 2. Границы (scope)

**Входит:** распознавание источника идеи → вызов нужного готового скилла → (где нужно) чекпойнт выбора → делегирование производства в `reels-script`/`carousel-script` → финальный артефакт + строка-трасса + «Допущения».

**НЕ входит:** дублирование логики суб-скиллов (только маршрутизация); автопостинг/доставка (слой Hermes/рантайма); новый `.env`/интеграции (делегирует суб-скиллам с их конфигом); персистентный контент-календарь/планировщик; сам агент Hermes и Telegram-бот (отдельный проект — content-machine лишь готов к управлению им).

## 3. Источники (роутер по входу)

Машина распознаёт источник по форме/смыслу входа и маршрутизирует:

| Источник | Триггер входа | Делегат | Чекпойнт |
|---|---|---|---|
| **Своя тема/идея** | текст без ссылки: «сделай контент про X», «есть тезис: …» | сразу к производству | нет |
| **«Найди идею»** | «о чём снять», «найди тему про AI», «контент-план» | `ai-research` (режим `ideas`) | да: показать идеи → выбор |
| **Ссылка на Instagram-ролик** | `instagram.com/p|reel|reels/…` | `instagram-downloader` (референс) | нет (одиночный) |
| **Профиль конкурента / «разбери нишу»** | `@handle`, «что залетает у X» | `instagram-analytics` (`scout`, опц. `enrich`) | да: показать топ → выбор |

Если источник неоднозначен — ровно один уточняющий вопрос.

## 4. Производство (выбор формата)

- Машина **рекомендует** формат под суть идеи: **рилс** (`reels-script`) / **карусель** (`carousel-script`) / **оба** (кросс-формат из одного зерна) и фиксирует выбор в блоке «Допущения».
- Пользователь (или вызывающий драйвер) может **переопределить** («сделай карусель»).
- Ниша пользователя пробрасывается в суб-скилл для кросс-переноса.
- Делегирование — через механизм скиллов (агент вызывает суб-скилл); content-machine НЕ повторяет их логику.

## 5. Чекпойнты (контроль + экономия токенов)

- Ресёрч-путь и analytics-путь имеют чекпойнт «показать → выбрать → произвести» (несколько выборов допустимы: «1 и 3 в рилс», «топ-2»).
- Своя идея и одиночный референс — без чекпойнта, сразу к производству с рекомендацией формата в «Допущениях».

## 6. Готовность к интеграции (headless / Hermes)

content-machine спроектирован как **чистый контракт «инструкция на входе → артефакт на выходе»**, не зависящий от того, кто вызывает (человек в CLI, Telegram-бот, агент Hermes на сервере).

- **Правило headless:** если входная инструкция уже задаёт всё необходимое (источник + идея + формат) — чекпойнты НЕ срабатывают, машина идёт напрямую. Так автоматический драйвер (Hermes/бот) гоняет конвейер без человека, просто формулируя конкретную инструкцию.
- **Интерактивный режим:** если входных данных недостаточно — срабатывают чекпойнты §5 (человек за CLI выбирает).
- Один и тот же скилл, два режима; инфраструктуры Hermes/доставки внутри НЕТ — интеграция позже = тонкая обёртка (Hermes вызвал content-machine → получил артефакт → доставил), без переписывания скилла.

## 7. Выход

- Финальные артефакты от format-скиллов (сценарий и/или карусель — в том виде, как их выдают сами скиллы).
- Сверху — короткая **строка-трасса**: `источник → идея → формат(ы)`.
- Блок **«Допущения»**: что машина решила сама (источник, идея, формат, ниша) — чтобы пользователь/драйвер мог поправить.

## 8. Дисциплина и этика (наследует)

- **Дисциплина ответа** — как у всех скиллов хаба: без «думания вслух»; видимый результат — артефакт + трасса + допущения; без финального CTA к диалогу.
- **Анти-репост** — наследуется от downloader/analytics: референс → переупаковка, не дословный репост.

## 9. Структура и файлы (подход A — тонкий делегат)

```
skills/content-machine/
├── SKILL.md          # роутер источников → чекпойнты → делегирование format-скиллам, headless-правило, трасса, допущения
├── README.md         # где применять + примеры по каждому источнику
└── evals/evals.json  # кейсы на каждый источник + выбор формата + headless
```

- **Без `.env`** — делегирует суб-скиллам с их конфигурацией.
- **DRY** — никакой логики reels/carousel/research/downloader/analytics внутри; только маршрутизация и хэндоффы. Если маршрутов станет много — таблицу роутинга вынесем в `references/` (мостик к структуре «SKILL.md + references-роли»), без переписывания.

## 10. Мета-промпт (где применять)

content-machine процедурный, поэтому бóльшая часть `SKILL.md` — детерминированный плумбинг (вызовы суб-скиллов, чекпойнты, формат трассы): это **не** прогоняется через meta-prompt (как и у downloader/analytics).

Но «ядро суждения» оркестратора — *как машина распознаёт источник по входу и рекомендует формат под суть идеи* — **prompt-like**. На этапе сборки (writing-plans → subagent-driven) это судительное ядро прогоняем через скилл `meta-prompt` как **сырьё** (роль, правила распознавания, критерии рекомендации формата, анти-неоднозначность), затем адаптируем в `SKILL.md`. Плумбинг остаётся детерминированным. Маршрут рекурсии — по правилу CLAUDE.md хаба.

## 11. Non-goals (YAGNI)

- Не дублировать логику суб-скиллов (только маршрутизация).
- Не делать автопостинг/доставку (слой Hermes/рантайма).
- Не вводить новый `.env`/интеграции.
- Не строить контент-календарь/персистентный планировщик.
- Не встраивать Hermes/Telegram-бот (отдельный проект; здесь — только headless-готовность).
- Meta-prompt — только для судительного ядра, не для всего SKILL.md.
