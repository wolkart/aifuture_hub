# Дизайн скилла `instagram-analytics` (разведчик ниши/конкурентов)

Дата: 2026-06-18
Статус: утверждён, готов к плану реализации (writing-plans)

## 1. Цель

Тул-скилл: по профилю(ям) конкурента и/или ссылкам на ролики через **Apify** собирает посты/рилс с метриками, ранжирует «что залетело» и выгружает в **структурированную базу** (Markdown-индекс + карточки + CSV).

**Ключевое (вшито):** это **разведка для создания СВОЕГО контента** — найти, что заходит в нише, разобрать сценарии, переупаковать под себя. НЕ репост чужого. Симметрично `instagram-downloader`.

**Двухуровневость (вшита в суть):** сначала дёшево-массово собираем базу (Tier-1), потом точечно-выборочно копаем глубоко (Tier-2). Не транскрибируем всё подряд — только то, что пользователь отметил.

## 2. Границы (scope)

**Входит:** Apify-разведка профиля/ссылок → нормализация по схеме → ранжированная выгрузка в Markdown (индекс + карточки) и CSV; выборочное Tier-2 обогащение (транскрипт + разбор сценария); хэндофф на переупаковку.

**НЕ входит:** живая интеграция с Google Sheets / БД (будущий «писатель» по той же схеме); аналитика своего аккаунта и time-series дашборды; поиск по хештегу/ключевому слову (вход — только профиль/URL); авто-обогащение всего подряд (Tier-2 — только по выбору); сама персистентная «база вирусных сценариев» + агент Гермес (отдельный проект, подключится по той же схеме); дословный репост (только переупаковка); meta-prompt (скилл процедурный).

## 3. Два режима (оркестрация: scout → enrich)

### `scout` (Tier-1 — дёшево, массово)
1. Вход: профиль(и) `@handle`/URL и/или конкретные ссылки на посты/рилс.
2. Apify тянет элементы с метриками.
3. Нормализация по фиксированной схеме (§5).
4. Ранжирование по вовлечённости (дефолт `engagement_rate`, по убыванию).
5. Пишет/обновляет: `index.md` (ранжированная таблица всех элементов) + карточки-скелеты `cards/<id>.md` + `export.csv`.
6. Токены почти не тратит; всегда работает (без ASR/LLM-разбора).

### `enrich` (Tier-2 — дорого, выборочно)
1. Вход: выборка — `top-N`, или список `#id`, или конкретные URL.
2. По каждому выбранному рилсу: скачать аудио из `video_url` → ASR (конфигурируемый) → LLM-разбор сценария (хук / тело-биты / CTA).
3. **Дописать** результат в существующую карточку и строку CSV; обновить `tier2_status` → `enriched`.
4. Грациозная деградация: ASR не настроен → транскрипт пуст, разбор делается по подписи; Tier-1-данные не ломаются.

## 4. Метод

- **Apify:** `APIFY_TOKEN` в `.env`; **актор конфигурируемый** (`APIFY_ACTOR`, дефолт — известный публичный IG-скрейпер). Вызов инлайн через `run-sync-get-dataset-items` (один запрос → сразу элементы датасета), парсинг адаптируется под фактическую форму ответа актора.
- **Ранжирование:** дефолт `engagement_rate = (likes + comments) / max(views, 1)` для рилс; конфигурируемо. Сортировка по убыванию.
- **ASR (Tier-2):** конфигурируемый `ASR_PROVIDER` (`openai` | `local` | `apify` | `none`) + грациозная деградация. Извлечение аудио из видео (`yt-dlp`/`ffmpeg` или прямой `video_url`).
- **Разбор сценария:** LLM раскладывает транскрипт на хук / тело-биты / CTA, опираясь на словарь виральности `reels-script` (общий вокабуляр паттернов).
- **Без отдельного исполняемого скрипта** — вызовы инлайн (`curl` к Apify API, `yt-dlp`/ASR-команда); вариативность вынесена в `references/`.

## 5. Выход (DB-ingestible, фиксированная схема)

```
<OUTPUT_DIR>/
├── index.md              # ранжированная таблица всех элементов (Tier-1) + ссылки на карточки
├── cards/<id>.md         # 1 карточка/элемент: frontmatter (вся схема) + секции; Tier-2 дописывает транскрипт+разбор
└── export.csv            # плоский CSV из тех же записей (для импорта в Sheets)
```

**Схема записи (поля):**
`id · author · url · type (reel/post/carousel) · date · caption · views · likes · comments · engagement_rate · video_url · hashtags · tier2_status (skeleton|enriched) · transcript · scenario{hook, body_beats, cta} · notes`

Эта схема — **единый контракт**: из неё рендерятся и Markdown, и CSV; она же — будущий контракт для Sheets/БД-писателя и веб-интерфейса пользователя.

- **`OUTPUT_DIR`:** дефолт — папка внутри проекта (`./instagram-analytics-output/`), переопределяется через `.env`. Obsidian-волт = просто папка: когда пользователь готов, указывает путь к волту в `OUTPUT_DIR` — без переписывания скилла.
- **Идемпотентность:** повторный `scout` обновляет существующие записи по `id`, не плодит дубли; `enrich` дописывает в существующую карточку/строку.

## 6. Хэндофф (обязательность — мягкая, в отличие от downloader)

- Обогащённый **рилс**-сценарий → `reels-script` (переупаковать под нишу пользователя).
- **Пост/карусель** конкурента → `carousel-script` (переупаковать под нишу).
- **База (Markdown) — самостоятельный артефакт** и сид для будущей «базы вирусных сценариев» (отдельный проект с Гермесом подключится по той же схеме).
- Этика: разведка публичных данных конкурента для своего контента; референс, не репост (как downloader). Соблюдение ToS — на стороне пользователя.

## 7. Ограничения

- **.env:** `APIFY_TOKEN`, `APIFY_ACTOR`, `ASR_PROVIDER`, `OPENAI_API_KEY` (если openai-ASR), `OUTPUT_DIR` (+ `.env.template`).
- **Дисциплина ответа** — как у всех скиллов хаба (без «думания вслух», без финального CTA к диалогу; видимый результат — сводка выгрузки + пути к файлам + при `enrich` краткий разбор).
- **Конкретика вместо абстракций**; не выдумывать метрики/цифры — только то, что реально вернул Apify; нет данных → поле пусто/помечено.

## 9. Non-goals (YAGNI)

- Не делать живой Google Sheets/БД-писатель (будущий апгрейд по той же схеме).
- Не делать аналитику своего аккаунта / time-series.
- Не делать поиск по хештегу/нише (вход — только профиль/URL).
- Не авто-обогащать всё подряд (Tier-2 — только по явному выбору).
- Не строить персистентную «базу вирусных сценариев» + Гермес (отдельный проект).
- Не репостить дословно (только переупаковка через reels-script/carousel-script).
- Не использовать meta-prompt (скилл процедурный).
